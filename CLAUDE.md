# CLAUDE.md

LLM 코딩 실수를 줄이기 위한 행동 지침. Andrej Karpathy의 원칙을 기반으로 작성되었으며, 퀀트 트레이딩 프로젝트에 맞게 조정되었습니다.

**트레이드오프:** 이 지침은 속도보다 신중함을 우선시합니다. 사소한 작업에는 판단을 사용하세요.

---

## 1. 코딩 전에 생각하기

**추측하지 말고, 혼란을 숨기지 말고, 트레이드오프를 표면화하라.**

구현하기 전에:
- 가정을 명시적으로 밝히세요. 불확실하면 물어보세요.
- 여러 해석이 가능하면 제시하세요 — 조용히 선택하지 마세요.
- 더 단순한 접근법이 있으면 말하세요. 필요할 때 반론을 제기하세요.
- 뭔가 불명확하면 멈추세요. 무엇이 헷갈리는지 명시하고 질문하세요.

**퀀트 트레이딩 맥락에서:**
- 전략 로직이 불분명하면 수익/손실 방향을 먼저 확인하세요.
- 데이터 소스나 타임존이 명시되지 않으면 반드시 물어보세요.
- 백테스트와 실거래 코드는 명확히 구분되어야 합니다.

---

## 2. 단순성 우선

**요청한 것만 해결하는 최소 코드. 추측성 코드는 없다.**

- 요청받지 않은 기능은 추가하지 마세요.
- 단일 사용 코드에는 추상화하지 마세요.
- 요청받지 않은 유연성이나 설정 가능성은 없습니다.
- 불가능한 시나리오에 대한 에러 핸들링은 없습니다.
- 200줄로 쓴 코드가 50줄이 될 수 있다면, 다시 작성하세요.

스스로에게 물어보세요: "시니어 엔지니어가 이것이 지나치게 복잡하다고 말할까?" 그렇다면 단순화하세요.

---

## 3. 수술적 변경

**필요한 것만 건드리세요. 자신이 만든 것만 정리하세요.**

기존 코드를 편집할 때:
- 인접한 코드, 주석, 포맷을 "개선"하지 마세요.
- 깨지지 않은 것은 리팩터링하지 마세요.
- 다르게 하고 싶더라도 기존 스타일을 맞추세요.
- 관련 없는 죽은 코드를 발견하면 언급은 하되 삭제하지 마세요.

변경으로 인해 고아가 된 것들:
- **당신의** 변경이 사용하지 않게 만든 import/변수/함수를 제거하세요.
- 요청받지 않으면 기존 죽은 코드는 제거하지 마세요.

테스트: 변경된 모든 줄은 사용자의 요청으로 직접 추적되어야 합니다.

---

## 4. 목표 주도 실행

**성공 기준을 정의하고. 검증될 때까지 반복하라.**

작업을 검증 가능한 목표로 변환하세요:
- "검증 추가" → "잘못된 입력에 대한 테스트 작성, 그런 다음 통과시키기"
- "버그 수정" → "재현하는 테스트 작성, 그런 다음 통과시키기"
- "X 리팩터링" → "전후로 테스트가 통과하는지 확인"

다단계 작업의 경우 간략한 계획을 제시하세요:
```
1. [단계] → 검증: [확인사항]
2. [단계] → 검증: [확인사항]
3. [단계] → 검증: [확인사항]
```

**퀀트 트레이딩 맥락에서:**
- 전략 변경 시 백테스트 결과(샤프 비율, MDD, 연간 수익률)를 검증 기준으로 포함하세요.
- 데이터 파이프라인 변경 시 데이터 품질 검사를 포함하세요.
- 실거래 연결 코드는 반드시 페이퍼 트레이드로 먼저 검증하세요.

## 5. 퀀트 트레이딩 특화 지침

**금융 코드는 일반 코드보다 더 높은 정확성 기준이 필요합니다.**

### 데이터 무결성
- 타임존을 항상 명시적으로 처리하세요 (KST, UTC 혼용 주의).
- look-ahead bias를 방지하세요 — 미래 데이터가 현재 결정에 영향을 주면 안 됩니다.
- NaN/무한값 처리를 명시적으로 하세요.

### 주문 및 포지션
- 매수/매도 방향은 항상 명시적인 상수나 열거형을 사용하세요.
- 포지션 크기와 레버리지는 항상 검사하세요.
- 실거래 API 호출 전에 주문 파라미터를 검증하세요.

### 성능 계산
- 수익률 계산 시 수수료와 슬리피지를 포함하세요.
- 샤프 비율 계산 시 연환산 기준을 명시하세요 (일별: ×√252).
- 백테스트 결과를 과적합 없이 해석하세요 — OOS(Out-of-Sample) 검증을 권장하세요.
